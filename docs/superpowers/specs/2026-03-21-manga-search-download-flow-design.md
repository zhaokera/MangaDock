# 漫画搜索后选章节下载设计文档

> **For agentic workers:** 先完成本文档评审，再进入实现计划与编码。

**Goal:** 在 `/manga` 页面保留“直接粘贴章节链接下载”的同时，新增“先搜索漫画作品，再进入作品详情选择章节，最后确认下载”的工作流。

**Architecture:** 前端在现有漫画页内新增漫画搜索、作品详情、章节多选和确认下载 4 个状态层；后端新增漫画搜索与章节列表接口；真正下载仍复用现有批量下载能力，把已选章节转换成多个章节 URL 后统一提交。

**Tech Stack:**
- React 18
- TypeScript
- Vite
- FastAPI
- 现有漫画爬虫与下载接口

---

## 背景与问题

当前漫画页只有“粘贴章节链接下载”这一条入口。这个模式对已经拿到章节链接的用户足够直接，但对只知道作品名的用户不友好，主要问题是：

- 用户必须先离开站点去找目标漫画链接
- 当用户只知道作品名时，当前漫画页没有可用入口
- 视频页已经支持“先搜索再下载”，而漫画页在工作流完整性上明显更弱
- 即使后端已有平台下载能力，前端也没有把“作品 -> 章节 -> 下载”这条路径组织出来

本次目标不是替换当前链接下载，而是在漫画页补齐搜索型工作流。

## 用户确认过的设计决策

- 保留当前漫画页的链接下载能力，不移除
- 漫画搜索范围只覆盖当前已支持漫画下载的平台
- 搜索命中后先进入作品详情，再选择章节
- 章节选择在漫画页内原地展开，不跳独立详情页，不用弹窗
- 章节支持多选
- 点击下载后先进入一个确认步骤，再真正提交下载
- 真正提交时沿用现有批量下载逻辑创建多个独立任务

## 用户流程

漫画页将同时提供两条入口：

### 入口 A：直接粘贴章节链接下载

- 继续保留现有 `UrlInput`
- 适合已经拿到章节链接的用户
- 保持当前单条下载与批量链接下载能力

### 入口 B：先搜索作品，再选章节下载

建议流程如下：

1. 用户输入漫画名称
2. 前端按漫画平台范围发起搜索
3. 返回作品结果列表
4. 用户点击某一部作品
5. 页面原地展开该作品详情与章节列表
6. 用户多选章节
7. 用户点击“下载所选章节”
8. 页面进入确认区，展示作品名、平台名、已选章节数量和章节标题摘要
9. 用户确认后，前端把章节 URL 列表统一提交给批量下载接口
10. 后端创建多个独立下载任务，前端沿用当前任务提示与历史记录机制反馈结果

## 页面结构调整

`/manga` 页面建议调整为以下顺序：

1. 漫画页 Hero
2. 漫画平台列表
3. 漫画搜索区
4. 搜索结果列表
5. 作品详情与章节选择区
6. 确认下载区
7. 原有链接输入区
8. 漫画下载进度
9. 漫画历史记录
10. 使用指南

这样搜索型路径放在主入口位置，链接下载仍然可见但退居辅助入口，不会打断已有用户。

## 前端组件与状态设计

### 页面状态

`MangaPage` 需要新增 3 组页面级状态：

- `searchState`
  - 搜索关键词
  - 平台筛选
  - 搜索结果
  - 搜索中状态
  - 搜索错误状态
- `selectedSeries`
  - 当前选中的作品
  - 作品详情加载中状态
  - 章节列表
  - 章节列表加载错误
- `chapterSelection`
  - 已选章节 ID 或 URL 集合
  - 是否显示确认区
  - 确认区摘要信息

下载中状态和当前任务状态继续沿用现有漫画页逻辑，但搜索型提交流最终也应走批量下载路径。

### 组件边界

建议新增或拆出以下边界：

- `MangaSearchInput`
  - 负责关键词输入、漫画平台筛选、触发搜索
- `MangaSearchResults`
  - 负责展示作品结果列表
  - 点击结果后通知页面层加载作品详情
- `MangaChapterPicker`
  - 负责章节列表展示、多选、全选、反选
- `MangaDownloadConfirm`
  - 负责展示已选章节摘要与最终确认按钮

如果希望减少组件数量，也可以先把 `MangaSearchResults` 与 `MangaChapterPicker` 作为漫画页内的子区块实现，但状态边界仍应保持分明。

### 与现有组件的关系

- `UrlInput` 保持存在，不与搜索区混合
- `DownloadProgress` 和 `History` 继续复用
- 现有 `SearchInput` 不建议直接硬套视频搜索逻辑
  - 可以抽共用样式壳
  - 但漫画搜索应走独立 API 和独立结果模型

## 后端接口设计

### 1. 漫画搜索接口

建议新增独立漫画搜索接口，而不是继续把漫画搜索塞进现有视频搜索接口。

建议形态：

- `GET /api/search/manga`

输入：

- `keyword`
- `platform`
- `limit`

输出：

- 漫画作品列表
- 每项至少包含：
  - `title`
  - `platform`
  - `platform_display`
  - `url`
  - 可选封面或简介字段

### 2. 漫画章节列表接口

建议新增章节列表接口：

- `GET /api/manga/chapters`

输入：

- `url` 或平台内作品 ID

输出：

- 作品基本信息
- 章节列表
- 每个章节至少包含：
  - `title`
  - `url`
  - 排序值或原始顺序

### 3. 下载提交接口

下载提交不新增复杂任务模型。

前端在确认后：

- 把已选章节转换成 URL 列表
- 统一调用现有批量下载接口

这样可以最小化后端改动，也能保证历史记录、进度流、任务落库继续沿用当前逻辑。

## 平台与范围约束

漫画搜索只针对当前已经支持漫画下载的平台。

这意味着：

- 搜索平台列表与漫画页平台列表保持一致
- 不因为“可搜索但不可下载”引入新平台
- 若某个平台当前只支持直链下载、不支持作品搜索，需要在实现时明确排除，或者为它补上最小搜索能力后再纳入

本次范围不包含“所有理论上的漫画平台搜索聚合”，只包含当前系统内实际可下载的平台。

## 确认下载区行为

确认区是本次交互的关键保护层，不能省略。

建议确认区展示：

- 作品名
- 平台名
- 已选章节数
- 章节标题预览
- 返回修改选择按钮
- 最终确认下载按钮

行为要求：

- 未选择任何章节时，不允许进入有效确认
- 确认区打开后，用户仍可返回章节区重新勾选
- 点击最终确认后，才真正调用批量下载接口
- 提交过程中应有明确 pending 状态，避免重复提交

## 错误处理

### 搜索层

- 无结果：提示“未找到匹配漫画”
- 搜索失败：提示具体失败信息，并允许直接重试

### 详情层

- 作品详情加载失败时，不清空搜索结果列表
- 章节列表加载失败时，只在详情区报错，允许用户切换其他作品

### 选择层

- 未选章节时点击下载，直接提示先选择章节
- 如果章节 URL 缺失或不可用，不进入确认步骤

### 提交层

- 批量提交失败时，保留当前已选章节与确认区状态，允许重新提交
- 批量提交部分成功时，沿用当前批量下载结果提示风格，明确成功数与失败数

## 测试要求

### 前端测试

至少覆盖：

- 漫画页新增搜索区，同时保留链接输入区
- 搜索结果点击后原地展示章节列表
- 支持多选章节
- 点击下载时先显示确认区，而不是直接提交
- 点击最终确认后才调用批量下载接口

### 后端测试

至少覆盖：

- 漫画搜索结果标准化
- 章节列表解析
- 已选章节 URL 列表的提交输入
- 漫画搜索平台过滤范围正确

### 手工验证

- 通过漫画名称可以搜到作品
- 选中作品后能看到章节列表
- 多选章节后可进入确认区
- 确认后能创建下载任务
- 原有漫画链接下载能力不受影响
