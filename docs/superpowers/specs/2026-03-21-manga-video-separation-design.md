# 漫画页与视频页分离设计文档

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 将当前混合的下载首页拆成独立的漫画页和视频页，避免用户再依赖 tag 才能理解当前操作对象。

**Architecture:** 前端增加明确的页面级分流，默认进入 `/manga`，并通过导航切换到 `/video`。漫画页只承载漫画链接下载，视频页只承载视频搜索和视频链接下载；下载进度、平台展示和历史记录也按页面类型分流显示。

**Tech Stack:**
- React 18
- TypeScript
- Vite
- 现有 FastAPI 接口（`/api/platforms`、`/api/download`、`/api/search`、`/api/history`、SSE 进度接口）

---

## 背景与问题

当前前端将漫画下载、视频搜索和视频下载放在同一套页面结构里，只通过 tab 或 tag 做弱区分。这个结构的问题是：

- 用户进入页面后，无法第一时间确认自己当前处于“漫画工作流”还是“视频工作流”
- 漫画和视频的主操作入口混在一起，页面语义不清晰
- 历史记录和下载进度没有页面级归属，用户需要自己判断结果属于哪一类内容
- 继续增加功能时，漫画和视频会进一步挤压同一页面，复杂度会持续上升

目标不是换样式，而是把信息架构拆开。

## 用户确认过的设计决策

- 采用两个独立页面，而不是单页双 tab 或单页上下双专区
- 首页不做单独分流页，应用默认落在漫画页
- 视频页同时保留“按名称搜索下载”和“粘贴视频链接下载”
- 历史记录和下载进度按页面类型拆开，各页只显示自己的内容

## 路由与导航

### 路由

- `/` 重定向到 `/manga`
- `/manga` 作为漫画下载页
- `/video` 作为视频下载页

### 导航行为

- 顶部导航明确提供 `漫画下载` 和 `视频下载` 两个入口
- 当前页导航项应有明确激活态，不能继续依赖平台 tag 提醒用户所在上下文
- 页面切换后，整页 hero、平台列表、主操作区、历史、帮助文案都应随页面类型完全切换

## 页面结构

### `/manga`

漫画页只承载漫画下载相关内容，推荐顺序如下：

1. 漫画页 Hero
2. 漫画平台列表
3. 漫画链接输入区
4. 漫画下载进度
5. 漫画历史记录
6. 漫画帮助说明

#### Hero

- 文案只描述漫画下载，不出现视频、搜索、番剧等表述
- 首屏应让用户立刻理解“这里是漫画页”

#### 平台列表

- 只显示漫画平台
- 不再展示视频平台，避免造成平台能力误判

#### 主操作区

- 保留当前单条下载和批量下载两种模式
- 输入说明、错误提示、示例链接都只围绕漫画链接
- 若用户粘贴的是视频链接，应直接提示这是视频内容，并引导切换到视频页

#### 下载进度

- 只展示漫画任务的当前进度
- 视频任务不应出现在漫画页内

#### 历史记录

- 只展示漫画历史
- 记录项的标题、章节、平台等信息按漫画语义展示

### `/video`

视频页只承载视频搜索与视频下载相关内容，推荐顺序如下：

1. 视频页 Hero
2. 视频平台列表
3. 视频搜索区
4. 搜索结果列表
5. 视频链接输入区
6. 视频下载进度
7. 视频历史记录

#### Hero

- 文案明确支持“搜索下载”和“链接下载”
- 首屏应清楚告诉用户这里是视频页

#### 平台列表

- 只显示视频平台
- 漫画平台不出现在视频页

#### 搜索区

- 搜索能力只存在于视频页
- 保留平台筛选能力，但筛选项只包含视频平台
- 搜索结果状态不应泄漏到漫画页

#### 视频链接输入区

- 视频页保留直链下载能力
- 若用户输入漫画链接，应提示切换到漫画页，而不是尝试下载或给出模糊错误

#### 下载进度

- 只展示视频任务的当前进度

#### 历史记录

- 只展示视频历史
- 记录项在视觉和文案上按视频语义展示

## 数据与状态设计

### 平台数据

- 继续复用 `/api/platforms`
- 前端根据平台名映射出 `manga` 与 `video` 两类
- 页面内部不直接消费“全量平台”，而是消费已经过滤好的 `mangaPlatforms` 或 `videoPlatforms`

这样可以避免子组件再次重复判断类型，也能确保页面不会意外渲染错误平台。

### 当前任务状态

- 不能继续由单一 `currentTask` 无条件驱动全局页面渲染
- 当前下载进度至少要按内容类型区分为漫画任务和视频任务
- 页面仅订阅和展示与自身类型对应的任务状态

如果短期内仍共享某些下载逻辑，页面层也必须在展示阶段进行类型隔离，不能让漫画页显示视频任务卡片。

### 历史记录分流

- 继续复用 `/api/history`
- 历史记录在前端按平台映射为漫画或视频
- 漫画页显示过滤后的漫画历史，视频页显示过滤后的视频历史

如果历史接口当前未显式返回内容类型，本次实现可以先通过平台名推断类型；是否新增后端字段不作为本次设计的硬性要求。

### 搜索状态

- 搜索关键词、筛选平台、结果列表、搜索错误态全部归属于视频页
- 漫画页不应保留任何搜索相关状态或组件引用

## 组件拆分建议

为避免再次把两类业务塞回一个大组件，建议页面层至少拆出以下边界：

- 路由壳层：负责导航与页面切换
- 漫画页容器：负责漫画平台、漫画下载、漫画任务状态、漫画历史
- 视频页容器：负责视频平台、视频搜索、视频直链下载、视频任务状态、视频历史
- 类型过滤工具或映射常量：集中维护平台到内容类型的映射

可以继续复用现有的输入组件和历史组件，但如果它们内部仍然强耦合“混合业务”，本次改造允许做针对性的拆分或参数化。

## 错误处理

### 页面类型错误

- 在漫画页输入视频链接时，提示“当前是漫画下载页，请切换到视频下载”
- 在视频页输入漫画链接时，提示“当前是视频下载页，请切换到漫画下载”

这种提示必须是明确的页面级指引，而不是笼统地说“链接无效”。

### 空状态

- 漫画历史为空时，显示漫画语义的空状态
- 视频历史为空时，显示视频语义的空状态
- 视频搜索无结果时，显示视频搜索语义的空状态

### 失败状态

- 下载失败继续在所属页面内反馈
- 搜索失败只在视频页反馈
- 不应再出现一类任务的错误信息跑到另一类页面中

## 视觉与文案约束

- 页面标题、Hero、副标题、按钮和帮助文案必须直接体现页面类型
- 不能再依赖 tag 或平台徽标作为唯一的类型提示
- 漫画页与视频页的首屏应有明显不同的语言和模块顺序，让用户一眼知道所在页面

## 实施边界

### 本次范围内

- 前端路由拆分
- 顶部导航拆分
- 漫画页和视频页页面结构拆分
- 平台、历史、下载进度按类型过滤显示
- 视频页保留搜索和直链下载
- 页面类型错误提示

### 本次范围外

- 后端接口大规模重构
- 新增完全独立的漫画搜索能力
- 历史记录数据库结构调整
- 全量视觉重做或品牌重设计

## 验证要求

### 手工验证

- 访问 `/` 时自动进入 `/manga`
- 导航可正确切换到 `/video`
- 漫画页只显示漫画平台和漫画下载能力
- 视频页只显示视频平台、视频搜索和视频链接下载能力
- 在错误页面输入另一类型链接时，出现明确引导提示
- 下载进度与历史记录在两个页面中不会串型

### 自动化验证

如果前端已有测试基础设施，本次至少覆盖以下行为：

- 路由切换后页面模块显示正确
- 平台列表按漫画/视频正确分流
- 历史记录按漫画/视频正确分流
- 视频页保留搜索模块，漫画页不渲染搜索模块

如果当前前端没有测试基础设施，本次可以先完成手工验证并在实现计划里补充测试建设步骤，但实现方案必须保留清晰的可测试边界。
